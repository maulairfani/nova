# backend/ — Nova's Backend API

FastAPI service implementing the Chat Endpoint and ReAct Agent (TDD §5.2).
Read [`documents/technical-design-document.md`](../documents/technical-design-document.md)
and the ADRs under `documents/adr/` before making non-trivial changes here —
they're the source of truth for *why* this is shaped this way.

## Structure

```
app/
  main.py              FastAPI app + lifespan (opens the Postgres checkpointer once)
  core/
    config.py           pydantic-settings, reads .env
    security.py          password hashing (bcrypt) + JWT issue/verify (ADR-0021)
    db.py                async SQLAlchemy engine/session for querying nova_core at runtime
    storage.py            MinIO client + bucket names (mcn-tv/mcn-plus/mcn-news, plus
                          CHARTS_BUCKET=nova-charts) - a small deliberate duplicate of
                          worker/minio_client.py's naming, used by document_service.py's
                          upload/delete/content and chart_service.py's read
    vectorstore.py         Qdrant client + collection names, same duplication reasoning -
                          used only by document_service.py's delete (removing a document
                          must also remove its points, or Nova would keep citing "deleted"
                          content)
  models/               SQLAlchemy models for nova_core's identity/access schema (ADR-0021),
                        one module per entity (base.py, user.py, business_unit.py,
                        conversation.py, document.py) — __init__.py re-exports so
                        `from app.models import X` is unaffected by which module defines X.
                        Source of truth for alembic/'s autogenerated migrations, NOT for
                        LangGraph's own checkpoint tables (those are checkpointer.py's).
                        conversation.py's Conversation is sidebar metadata (title, user_id,
                        updated_at) for a chat thread, deliberately separate from the
                        checkpointer's own tables (same Document/ADR-0022 split): deleting
                        a conversation must delete both this row and the matching
                        checkpointer thread (conversation_service.py does both, in that
                        order)
  schemas/auth.py        LoginRequest/LoginResponse
  repositories/          Data-access layer — one module per aggregate (user, conversation,
                        document), each a thin set of functions wrapping a SQLAlchemy
                        session (select/get/create/delete, commit included). No business
                        logic, authorization, or calls to MinIO/Qdrant/the checkpointer here
                        — that's services/. Endpoints never import repositories directly.
  services/              Business-logic layer, one module per endpoint's domain
                        (auth_service, conversation_service, chat_service,
                        document_service, chart_service) — owns authorization checks
                        (document_service's _require_view/_require_manage), orchestration
                        across repositories + MinIO/Qdrant/the checkpointer/the agent, and
                        anything an endpoint would otherwise have needed to know about
                        persistence. Endpoints call services, never repositories or
                        SQLAlchemy directly.
  api/v1/
    deps.py              verifies a request's JWT, derives the X-Nova-* header shape every
                        MCP server's AuthContext expects (replaces the old unverified
                        header pass-through)
    endpoints/           thin HTTP/SSE adapters only — parse the request, call the matching
                        service, shape the response. No SQLAlchemy, MinIO, or Qdrant calls
                        live in this directory (see services/ above for where that moved).
      auth.py             POST /auth/login — no signup, accounts are seeded (seed_users.py)
      chat.py              Chat Endpoint — SSE (ADR-0017), delegates to chat_service for the
                          event stream and conversation_service for the Conversation upsert
                          (title on first message, updated_at bump on later ones)
      conversations.py     Sidebar list/rename/delete + read-back of a thread's message
                          history, delegates to conversation_service (which reads the
                          checkpointer's stored state directly rather than rebuilding the
                          full agent), including each assistant turn's tool-call steps
                          reconstructed from its AIMessage.tool_calls, and any charts
                          reconstructed from its ToolMessage results (ADR-0026)
      documents.py          Manage Documents Endpoint - list/upload/delete knowledge base
                          source files, plus a read-only content endpoint
                          (GET /documents/{id}/content, streams raw bytes from MinIO for
                          the frontend's document preview), delegates to document_service
                          (the only place backend/ talks to MinIO/Qdrant, see below)
      charts.py             Chart Endpoint (ADR-0026) - GET /charts/{chart_id}, delegates to
                          chart_service, which streams a chart image the Chart Generation
                          Tool (mcp-shared) rendered and uploaded to MinIO's nova-charts
                          bucket. Any authenticated caller may fetch any chart_id - charts
                          aren't business-unit-scoped data
  agent/
    graph.py            create_agent wiring — the ReAct Agent itself
    llm.py              LLM Client — OpenRouter (ADR-0015/0018)
    mcp_client.py        MCP Client — MultiServerMCPClient + Redis tool-result cache wrapper
    checkpointer.py       Postgres checkpointer factory (nova_core)
    cache.py             Cache Client (Redis, pickle-serialized — see note below)
    prompts.py
alembic/                Migrations for models/'s schema (ADR-0021) — generated via
                        `alembic revision --autogenerate`, not hand-written like the
                        per-business-unit MCP servers' own alembic/ (ADR-0016). env.py's
                        include_object filter excludes LangGraph's checkpoint-owned tables
                        so autogenerate never proposes dropping them.
seed_users.py           One-off dummy-user seed (ADR-0021) — see SEED_USERS.md for the
                        actual accounts/passwords/roles it creates.
```

**Layering convention (endpoints → services → repositories/external clients):**
endpoints/ only parse requests and shape responses; services/ hold every
business rule (authorization, title-derivation, event-stream construction)
and are the only layer allowed to call more than one repository or an
external client (MinIO/Qdrant/the checkpointer/the agent) in the same
operation; repositories/ are pure data access for `nova_core` (no
authorization, no non-Postgres I/O). Chosen to match this codebase's
`fastapi-templates` skill guidance rather than freestyling a shape once
the endpoint count grew past a handful of simple CRUD routes.

## Real authentication (ADR-0021)

Login is `POST /api/v1/auth/login` (email+password against `users`, no
signup — accounts are seeded via `seed_users.py`, see `SEED_USERS.md`),
issuing a JWT whose claims embed the caller's `business_units` + per-unit
`role_code` pairs at login time (stateless — a membership change only
takes effect on next login). `api/v1/deps.py`'s `get_auth_headers`
verifies that JWT on every request and derives the `X-Nova-*` header shape
each MCP server's `AuthContext` expects — this replaces the old phase-1
behavior of forwarding whatever headers the Frontend happened to send,
unverified.

Two real bugs surfaced in `mcp_client.py` while wiring this up, both now
covered by regression tests (`tests/test_mcp_client.py`):

1. An identity with no real business-unit claim (e.g. an MCN Group
   corporate employee, ADR-0021's `group`/`employee` membership) used to
   get **zero tools at all**, including web search — `get_tools_for_identity`
   returned early before the Shared MCP Server was even added to the
   connection list. Fixed by always including the shared server; its own
   auth check is the real gate for a truly anonymous caller.
2. `group_admin` (the `group`/`admin` tier, bridged to the `"group_admin"`
   role string for backward compatibility with each unit's existing
   `ALLOWED_ROLES` check) used to get **no cross-unit access at all** even
   though each unit's own auth check already allowed it in — the backend's
   own server-selection logic only connected to servers matching the
   caller's literal claimed business units, never attempting a unit the
   caller hadn't explicitly claimed. Fixed by connecting to every business
   unit's server when `group_admin` is present, regardless of claimed
   units.

## Tool-call steps (chat.py's SSE stream)

`chat.py`'s `event_stream()` now also forwards `on_tool_start`/`on_tool_end`
events from `agent.astream_events(...)` as new SSE event types
(`event: tool_start` / `event: tool_end`, keyed by LangGraph's `run_id` so
the frontend can match a start to its end even with multiple tool calls
in one turn) alongside the existing token-delta `data:` frames. Each
tool's business-unit-prefixed name (e.g. `tv_kb_search`, mcp_client.py's
`_wrap_with_cache`) is mapped to a human-readable `{type, label}` step by
`app/agent/tool_labels.py` - shared by both the live stream and
`conversations.py`'s history reconstruction (walking a reloaded thread's
raw `AIMessage.tool_calls` to rebuild the same steps for a past turn), so
a freshly-streamed answer and a reloaded one show identical step labels.

## Sources panel + inline citations (2026-07-18)

`kb_search`/`web_search` tool results are extracted into citation cards
for the frontend's Sources panel + inline 【Title】 badges, the same
shared-helper pattern `parse_chart_result`/`map_tool_step` already
established for the tool trace and chart features:

- **`app/agent/tool_labels.py`**: `parse_citations(tool_name, raw_content)`
  pulls `{type, title, snippet, unit?, source_document?, url?}` cards out
  of a `kb_search`/`web_search` ToolMessage - chunk-level (one call can
  return several). `sql_analytics` has no "source" in this sense and is
  deliberately excluded (its transparency artifact is the generated SQL,
  already shown via the tool-call trace, not a citable document/page).
  `merge_citations(citations, new_raw)` then dedupes by
  `(type, source_document or url)` into a turn's running list - the same
  document cited via several chunks, or re-retrieved by a later tool
  call, gets one entry. **A citation's number is its 1-based position in
  this list** - the LLM never sees or tracks this number itself (see
  SYSTEM_PROMPT below); it cites by title, which the frontend matches
  back to an entry.
- **`app/agent/prompts.py`**: SYSTEM_PROMPT instructs the model to mark a
  cited fact with the source's *exact title* wrapped in full-width
  brackets (`【Ad Slot Booking SOP — MCN TV】`) - chosen over an abstract
  `[1]`/`[[1]]` scheme because the model copies text it already sees
  verbatim in the tool result, rather than tracking a number it's never
  shown. Verified directly against a live chat request: `gpt-5.4-nano`
  reliably reproduces the exact title.
- **`chat.py`/`chat_service.py`**: a new `event: citations` SSE event,
  emitted after `tool_end` whenever a tool call's citations extend the
  turn's running (deduped) list - sends the *whole* current list each
  time (small, simpler than an incremental diff) so the frontend just
  replaces its stored array.
- **`schemas/conversations.py`**/`conversation_service.py`: `CitationOut`
  + `MessageOut.citations`, reconstructed from a reloaded thread's stored
  `ToolMessage`s via the same `parse_citations`/`merge_citations` pair,
  buffered/reset on the same lifecycle as `pending_steps`/`pending_charts`
  - a live-streamed answer and a reloaded one number citations
  identically.

Verified end-to-end against the live stack (not just unit-tested): a
real chat request's raw MCP response confirmed FastMCP wraps a tool's
list return as `structuredContent.result` (not the bare list, unlike a
dict-returning tool like `generate_chart`) - `parse_citations` was
written against that confirmed shape. A live `kb_search` and
`web_search` question both produced correct `event: citations` frames
and a model-emitted 【Title】 marker matching a real citation; a full
page reload of that conversation reconstructed identical citation
numbers and badges from history. `tests/test_tool_labels.py` covers
`parse_citations`/`merge_citations` against realistic MCP-wire-shaped
fixtures (kb dedup-by-source_document, web extraction, malformed-content
fallback).

**"View document" on a kb citation card** (frontend's `SourcesPanel.tsx`)
reuses the existing Manage Documents preview flow (`get_document_content`,
`_require_view`) rather than a raw MinIO URL (MinIO isn't public, and a
presigned URL would bypass this app's own auth). A kb citation only
carries `(unit, source_document)` from the Qdrant payload - resolving
that back to a `documents` row needed a new lookup:
`document_repository.get_by_unit_and_object_key` /
`document_service.find_by_source` / `GET /documents/lookup?business_unit=&object_key=`.
**`documents.object_key`'s format is not uniform** - Manage Documents'
own upload UUID-prefixes it (`{uuid4()}-{filename}`), while
`worker/seed_documents.py`'s direct-MinIO path does not - but an
**exact** `(business_unit_code, object_key)` match is still correct and
unambiguous either way: `worker/tasks.py` always writes a Qdrant chunk's
`source_document` payload field as the exact same string as whatever
`object_key` triggered that ingestion, and the pair is DB-unique
(`UniqueConstraint` on `Document`). Verified directly against the live
stack (not assumed): `curl`ed the new endpoint for a real document (200,
correct row) and a nonexistent one (404), then a full browser pass -
clicking "View document" on a citation card opens the real document's
content in the existing `DocumentPreviewModal`, stacked over the Sources
panel.

## Clickable reply options: quick-replies + multi-choice (2026-07-18)

`app/agent/prompts.py`'s `SYSTEM_PROMPT` gained instructions for two
fenced-code-block markers the frontend renders as clickable buttons instead
of plain code (`lib/NovaMarkdown.tsx`'s `pre` override on the frontend,
detected by language tag rather than a text regex): ` ```nova-quick-replies ```
(one option per line, clicking sends it immediately) and
` ```nova-multi-choice ``` ` (clicking toggles selection; the frontend
writes the joined selection into the existing composer rather than adding
a separate confirm button). This is pure prompt-text - no backend code
change beyond the prompt string, since the SSE stream already passes
through arbitrary markdown untouched. The prompt explicitly tells the
model the example option lines shown are just format examples to replace,
not literal content to reuse, to reduce the risk of a small model echoing
the example verbatim regardless of the actual conversation. See
`frontend/CLAUDE.md`'s matching section for the frontend implementation
and a real state-loss bug it surfaced (and fixed) in `NovaMarkdown.tsx`'s
memoization.

## Manage Documents (documents.py)

Real CRUD on the knowledge base's source files, backed by the same
infrastructure worker/'s ingestion pipeline already uses (ADR-0022) - not
a separate/duplicate ingestion path:

- **Upload** writes the file straight into the target business unit's
  MinIO bucket (`mcn-tv`/`mcn-plus`/`mcn-news`) under a
  `{uuid4()}-{filename}` object key. MinIO's existing webhook
  notification (already configured for the real pipeline) fires exactly
  as it would for any other upload, and `worker/`'s Celery task parses,
  embeds, and upserts it - this endpoint never touches Qdrant to *add*
  anything.
- Before writing to MinIO, the endpoint pre-creates the `Document` row
  itself (status `pending`, with the caller's chosen title) rather than
  waiting for `worker/`'s `insert_pending` to create one. `worker/db.py`'s
  `insert_pending` was changed to get-or-create by
  `(business_unit_code, object_key)` (a unique constraint added in
  migration `0004`) so it finds this pre-created row instead of inserting
  a duplicate, and `mark_ingested` only overwrites `title` with the
  parser's extracted title when the row is still using its `object_key`
  placeholder - a human-provided title is never clobbered. The legacy
  direct-MinIO-upload path (no pre-existing row, e.g. `seed_qdrant.py`)
  is unchanged: `insert_pending` still creates the row itself, title
  defaults to `object_key`, and gets replaced by the parsed title on
  success exactly as before.
- **Delete** removes all three: the document's points from its unit's
  Qdrant collection (filtered by the `source_document` payload field
  worker/ writes on ingest), the MinIO object, and the `documents` row -
  in that order, so a caller never sees a document listed as gone while
  Nova can still retrieve and cite its content.
- **Authorization** (documents.py's `_accessible_units`/`_can_manage`):
  viewing a unit's documents requires being a member of that unit (or
  `group_admin`, same scoping as `mcp_client.py`'s tool access - a plain
  `group` membership with no admin tier sees no unit's documents, same as
  it gets no unit's MCP tools today); uploading/deleting requires the
  `admin` tier within that specific unit, or `group_admin` for
  cross-unit management. This is the first real use of
  `business_unit_roles`' `admin` tier anywhere in the codebase - it was
  stored and forwarded since ADR-0021 but never enforced until now.

## Phase-1 simplifications still outstanding (documented, not silent)

- **A fresh `MultiServerMCPClient` per chat request.** `langchain-mcp-adapters`'
  HTTP headers are static per client instance (no per-call dynamic header
  injection as of this writing) — since identity varies per request, a new
  client is built per request rather than once at startup. Tool *schemas*
  don't depend on identity, so this only costs a cheap connection per
  request, not a re-architecture. Revisit if this becomes a real latency
  concern at higher traffic (not observed so far).
- **`business_unit_roles`' tiers (`finance`, `admin`) still aren't enforced
  by any MCP server.** The identity/access schema (ADR-0021) stores the
  tier, and it's embedded in the JWT and forwarded, but no unit's SQL
  Analytics Tool (`run_select`) actually checks it — every unit member
  currently gets the same data-query access regardless of tier. The
  `admin` tier *is* now enforced for Manage Documents (`documents.py`,
  above) — that's a separate authorization surface from the MCP servers'
  own tool-level checks, so this gap is narrower than before but not
  closed.
- **`get_tools_for_identity` only connects to the caller's claimed business
  unit's server(s)**, not every business unit unconditionally. Found this
  the hard way in phase 2: with 3 servers live, an unscoped tool list let
  the LLM attempt tools on a unit the caller wasn't authorized for, and
  that server's auth denial came back as an unhandled exception that
  crashed the whole SSE response rather than failing gracefully. Every
  server also exposes identically-named tools (`kb_search`,
  `sql_analytics`), so each tool's exposed name is prefixed with its
  business unit (e.g. `tv_kb_search`) — without that, the combined tool
  list would have name collisions and dispatch would be ambiguous about
  which server a call actually reaches.
- **The Shared MCP Server (`mcp-shared`, web search) is always included**,
  unlike `tv`/`plus`/`news` — it isn't owned by any single business unit,
  so it isn't filtered by the caller's claimed unit(s).
- **Tool calls that fail are caught and returned as content, not raised.**
  `langchain-mcp-adapters` converts *any* MCP tool error (not just an auth
  mismatch — an invalid API key, a rate limit, any downstream failure)
  into a raised exception, and LangGraph's default tool-error handling
  re-raises it, crashing the whole SSE response over one failed tool call.
  This surfaced concretely with the web search tool (an external API is
  far more likely to fail than our own servers). `_wrap_with_cache` now
  catches any exception from the underlying tool call and returns it as a
  string instead — the agent sees a failed tool result and can respond
  gracefully (per `prompts.py`'s existing instruction not to guess an
  answer), rather than the whole request crashing.
- **Tool-result cache uses `pickle`, not JSON.** MCP tool results returned
  by `langchain-mcp-adapters` can be a `(content, artifact)` tuple, and a
  JSON round-trip silently turns that into a list — a different shape than
  what LangChain's tool-calling code expects, which caused a real bug
  during testing (a cache hit returned a differently-typed result than a
  cache miss). `pickle` preserves the exact type. Safe here because this
  cache only ever stores data the app itself wrote (never user input) and
  isn't reachable outside the Docker network.

## Layered refactor to match `fastapi-templates` (2026-07-18)

`app/models.py` (one flat file for all 6 ORM classes) became `app/models/`
(one module per entity), and every endpoint's inline SQLAlchemy/MinIO/
Qdrant/checkpointer calls moved into a new `app/repositories/` (pure data
access) + `app/services/` (business logic, authorization, orchestration)
split — endpoints/ are now thin adapters only. This was a pre-existing gap
against `.claude/skills/fastapi-templates/SKILL.md`'s recommended project
layout (that skill has been present in this repo since before Q2
implementation started, but the endpoints were built directly against
SQLAlchemy/MinIO/Qdrant without the extra layers as the domain logic
stayed small — the gap only became worth closing once `documents.py` and
`conversations.py` had grown real authorization + orchestration logic
mixed into route handlers). No behavior change - verified via the full
unit suite (23/23, no regressions) and a live-stack pass covering every
touched endpoint (login, list conversations, usage, list documents, and a
real streamed chat message) after rebuilding `backend-api`.
